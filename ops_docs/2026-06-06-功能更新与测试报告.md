## 二、功能说明

### 2.1 矛盾知识入库仲裁（kb_service）

**策略**（`kb_conflict.py`）：

| 场景 | 行为 |
|------|------|
| 问题相似 + 答案相同 | `duplicate`，跳过入库 |
| 问题相似 + 答案矛盾，priority 不同 | 高 priority `supersede` 低 priority |
| priority 相同，**双方都有 date** | 新日期 supersede，旧日期 reject |
| priority 相同，date 无法裁决，**双方都有 version** | 高版本 supersede，低版本 reject |
| 以上均无法裁决 | `coexist`，双条保留，新条 priority +1 |

**阈值**：矛盾相似度 `0.80`，去重相似度 `0.90`。

**仲裁顺序**（仅当问题相似且答案不同时）：

```
priority 比较 → date 比较（双方都有） → version 比较（双方都有） → coexist
```

**检索排序**（入库后）：`priority ↓` → `version ↓` → `date ↓` → `相似度 ↓`。

---

#### 2.1.1 日期（date）与版本（version）如何取值？

| 字段 | 来源 | 说明 |
|------|------|------|
| `date` | frontmatter `kb_date` / `date`；或 API `metadata.date` | **不从正文解析**「2025年1月」等表述 |
| `version` | frontmatter `kb_version`；默认 `1`；supersede 时系统 `旧 version + 1` | 整数比较 |
| `priority` | frontmatter `kb_priority`；`ops_doc` 默认 `100`；`ticket` 源 `60` | 先于 date/version 裁决 |

**`parse_date` 可读字段**：`date`、`revision_date`、`updated_at`、`doc_date`（支持 `YYYY-MM-DD` 或 ISO 时间）。

**`both_have_dates` 规则**：新旧两条 **都必须** 有可解析的 `date` 才会进入日期比较；任一方无 `date` → 跳过日期裁决，改看 `version` 或 `coexist`。

---

#### 2.1.2 ingest 日期策略（2026-06-06 修订）

**旧逻辑（已废弃）**：Markdown 无 frontmatter 日期时，自动写入 **导入当天的本机日期**（`date.today()`）。

**问题**：早年写的文档若晚导入，会被标成「今天」，在日期仲裁中误当作最新文档，可能错误覆盖带明确旧日期的条目。

**新逻辑**（`ingest.py`）：

| 情况 | `metadata.date` |
|------|-----------------|
| frontmatter 有 `kb_date: 2026-06-06` | 写入 `2026-06-06` |
| 无 frontmatter 日期 | **不写入 `date` 字段** |

因此：

- 无 `date` 的文档 **不参与** 日期仲裁（不会因「导入日」被当成最新）。
- 需要按修订日覆盖的正式文档，应在 frontmatter **显式标注** `kb_date`。
- 工单飞轮 `/api/kb/add` 仍可通过 `metadata.date` 按需传入。

**frontmatter 示例**：

```yaml
---
kb_date: 2026-06-06
kb_version: 2
kb_priority: 100
---
```

**测试文档对照**：

| 场景 | 文件 | 期望 |
|------|------|------|
| 日期覆盖 | `scenario-b-date-old.md`（2025-01-01）→ `scenario-b-date-new.md`（2026-06-06） | 新日期 supersede |
| 版本覆盖 | `scenario-c-version-v1.md`（v1）→ `scenario-c-version-v2.md`（v2） | 高版本 supersede |

> **存量数据**：若库中条目曾被错误标为「导入当天」，不会自动修正；需用正确 `kb_date` **重新 ingest**，或 API 修正 metadata。

---

**文档入库**：`ingest.py` 支持 frontmatter（`kb_date` / `kb_version` / `kb_priority`）；运维文档默认 `priority=100`，`version` 默认 `1`。

**API**：`/api/kb/add` 响应含 `action`、`conflict`、`superseded_ids`。

**测试文档**：`ops_docs/conflict_test/`（8 个场景）；脚本 `kb_service/scripts/run_conflict_doc_tests.py`。

### 2.2 RAG 多答案与确定性直出（rag_ops.py）

**检索**：`get_kb_documents` 调用 8000 `/api/kb/search`，`sort_by_priority=true`，最多 3 条；**按答案文本去重**，避免重复 FAQ 占满槽位。

**回答路径**（优先级从高到低）：

```
意图分流（寒暄 / 会话回顾 / 知识库管理 / 违禁话题）
    ↓ 否
知识库 + knowledge.txt 检索（相似度过滤）
    ↓ 无可靠命中 → 拒答 + need_human
多条 KB 且答案不同且 top_score ≥ 0.60 → format_multi_kb_answer（不走 LLM）
单条或重复答案且 top_score ≥ 0.60 → format_direct_answer（不走 LLM）
    ↓ 否则
小模型 LLM 兜底（deepseek-r1:1.5b）
```

**多答案格式示例**：

```
关于「DNS解析失败怎么办」，知识库中有 2 条相关参考（按 priority 从高到低）：

【参考1·推荐】priority=101，相似度 65%
问题：并存测试：DNS故障怎么排查？
方案：方案B

【参考2·备选1】priority=100，相似度 67%
问题：并存测试：DNS故障怎么排查？
方案：方案A

——
多条参考并存时，默认优先采纳 priority 较高的方案；仍有疑问请转人工确认。
```

**上下文组装**：KB 去重后不足 3 条时，用 `knowledge.txt` 本地检索补足（阈值 0.60）。

---

### 2.3 拒答与相似度阈值

| 参数 | 当前值 | 说明 |
|------|--------|------|
| `MIN_KB_SCORE` | **0.60** | KB 命中、多答案直出、单条直出的最低相似度 |
| 本地 `knowledge.txt` 检索阈值 | **0.60** | Chroma `score_threshold` |
| kb_service RAG 过滤（参考） | 0.60 | `main.py` 内 `relevant_results` 过滤 |

**拒答文案**（`REFUSE_ANSWER`）：

> 抱歉，我暂时无法处理该问题。请点击转人工，运维工程师将稍后联系您。

**触发条件**：无上下文文档；KB top score < 0.60；违禁话题（天气、电影等）；非运维且无可靠命中。

**`need_human`**：`answer == REFUSE_ANSWER` 或拒答类「抱歉…无法处理」；**不再**因多答案文末「转人工确认」误触发。

### 2.4 意图分流（不走知识库检索）

| 类型 | 识别方式 | 行为 | need_human |
|------|----------|------|------------|
| 寒暄 | 精确匹配：你好、在吗、你是谁… | 固定运维助手自我介绍 | false |
| 会话回顾 | 正则：刚才/刚刚说了什么、你还记得… | 用 `history` 回顾；无历史则提示翻看聊天记录 | false |
| 知识库管理 | 正则：更新知识库、知识库入库… | 说明飞轮 / ingest / 联系管理员 | false |
| 违禁话题 | 关键词：天气、电影、游戏… | 拒答 | true |

**会话历史传递**：`Chat.jsx` → `8082 ChatController` → `8001 /api/rag` 的 `history` 字段（`role` + `content`）。

### 2.5 SSE 流式与前端显示

| 问题 | 原因 | 修复 |
|------|------|------|
| Markdown 粘成一行 | 8001 按 2 字切片 SSE；Java/前端 `trim()` 吃掉 `\n` | `api_server.py` **按行**推送；`ChatController` 行间补 `\n`；`Chat.jsx` 拼接保留换行 |

### 2.6 工单软删除联动（数据飞轮）

- 工单处理完成（resolve）：Java 调 `/api/kb/add`，ID 为 `kb_ticket_{工单号}`，语义去重。
- 工单删除：`TicketController` 调 `/api/kb/soft-delete`，按 `kbEntryId` 标记 `active=false`，检索时过滤。
- `TroubleTicket` 实体增加 `kbEntryId` 字段。

## 六、已知限制与演示注意

1. **会话记忆**：仅当前页面 `history` 有效，刷新或新开页面不保留；非持久化会话库。
2. **8082 history**：`ChatController` 已加 `history` 字段，需 **重新编译并重启 Java** 后，前端会话回顾才走完整链路。
3. **测试数据污染**：库中可能存在 `并存测试` 短文本、`CONFLICTTEST-*` 等条目，可能影响 DNS 等演示 Top 结果；演示前可清理。
4. **小模型兜底**：仅在无可靠 KB 命中且未命中意图分流时使用；弱匹配应已被 0.60 阈值拦截。
5. **Maven 构建**：本地 `settings.xml` 异常时需在 IDE 或修复 Maven 后重新打包 8082。
6. **日期 metadata**：正文里的日期文字不会自动入库；需 frontmatter `kb_date` 或 API 传入。无 date 的条目不参与日期仲裁；存量错误日期需重新 ingest 修正。

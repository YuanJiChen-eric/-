## 幻灯片 2 · 数据飞轮

### PPT 页面文字

**标题：数据飞轮 · 知识如何越来越多**

```
① 用户提问 → RAG 检索 8000 → 答不了 → 自动产生 pending 工单

② 运维处理工单（填写解决方案）
        ↓
   Java resolve → POST /api/kb/add
        ↓
   写入 kb_ticket_{工单号}  （语义去重，避免重复）

③ 下次类似提问 → 检索命中 → 系统能答

④ 删除工单 → POST /api/kb/soft-delete → active=false（软删除，检索自动忽略）
```

人工处理一次，知识库增长一次；删工单不留下脏数据。

---

### 讲述内容

> 这一页是 **数据飞轮**，也是课题里「处理完成后再自动完善知识库」在知识库的实现。
>
> 第一步，用户在前台提问，RAG 会去我的知识库 **检索**。如果匹配度不够、系统答不了，会自动产生 **待处理工单**。
>
> 第二步，运维在后台填写 **标准解决方案** 并提交。成员 B 的 Java 后端在处理工单时，会自动调用 `/api/kb/add` **接口**，把「用户问题 + 人工答案」写入向量库。每条工单对应稳定 ID `**kb_ticket_{工单号}`**，并做 **语义去重**，相似度很高的重复问题不会反复入库。
>
> 第三步，以后再遇到 **类似问题**，检索就能命中这条新知识，系统可以直接回答，这就是 **飞轮闭环**。
>
> 我们还做了 **生命周期管理**：如果管理员 **删除已处理工单**，会同步调用 `**/api/kb/soft-delete`**，把对应知识标记为无效，**检索时自动忽略**，避免删了工单、知识库里还留着过期答案。

## 幻灯片 3 · 技术原理与效果

### PPT 页面文字

**标题：检索原理 & 实现效果**

**原理**

1. 文档按标题切片 → 问题+答案成对入库
2. BGE-M3 向量化 → ChromaDB 余弦相似度检索
3. 返回 Top-K，score ≥ 0.60 才作为有效上下文

**效果（实测）**


| 测试问句      | Top1 相似度  | 说明                          |
| --------- | --------- | --------------------------- |
| 账号冻结怎么处理？ | **0.76**  | 命中课题示例（自助网址 + 热线）           |
| 知识总量      | **630 条** | 种子 FAQ + 文档切片               |
| 联调        | ✅         | resolve 入库 / delete 软删除 已验证 |


---

### 讲述内容

> 检索原理可以概括三句话：**先把运维文档切成问答块**；用 **BGE-M3** 转成向量存进 ChromaDB；用户提问时做 **相似度检索**，返回分数最高的若干条。我们约定 **相似度 0.6 以上** 才当作可靠上下文，减少误答。
>
> 效果上，用课题标准示例「**账号冻结怎么处理**」测试，Top1 相似度约 **0.76**，能命中自助网址和热线。知识库目前 **630 条**，覆盖账号、网络、数据库等常见运维场景。
>
> **端到端联调** 也已完成：工单处理会入库，删工单会软删除。 **检索和知识沉淀的内容到此介绍完毕。**
